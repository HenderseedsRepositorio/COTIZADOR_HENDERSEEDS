🔴 Problemas criticos

01
UX: Campos y lotes se crean con prompt()
El usuario necesita 3-4 prompts secuenciales para agregar un campo (nombre, cliente, localidad) y 3 mas para cada lote (nombre, hectareas, cultivo). En mobile esto es muy molesto: el teclado se abre/cierra por cada prompt, pierde contexto visual, no puede corregir campos previos. Lo mismo para editar.
UX CRITICO Reemplazar con modal inline
02
XSS: innerHTML con datos de usuario sin sanitizar
Lineas 612-640 (renderCampos), 808-882 (renderAllApps), 1407-1471 (renderOrdenes) usan innerHTML con nombres de campos, lotes, productos y aplicadores directamente interpolados. Un nombre como <img src=x onerror=alert(1)> ejecutaria codigo.
SEGURIDAD Escapar o usar textContent/DOM API
03
Datos: localStorage como unica fuente = riesgo de perdida total
Todo el trabajo del agronomo (campos, lotes, ordenes de toda la campana) esta en localStorage del navegador. Un borrado de cache, cambio de dispositivo o clear accidental pierde todo sin posibilidad de recuperacion.
DATOS Export/Import JSON + backup a Sheets
04
Analisis no usa multi-aplicacion
El tab Analisis (lineas 1488-1588) filtra por o.loteId y o.hectareas (campos legacy de la primera aplicacion) en lugar de iterar o.aplicaciones[]. Ordenes con multiples lotes solo contabilizan el primer lote.
BUG Iterar o.aplicaciones en renderAnalisis
05
WA message no refleja multi-aplicacion
_buildWAMsg() (linea 1595) usa o.loteId, o.hectareas y o.productos (flat). No muestra a que lote pertenece cada producto cuando hay multiples aplicaciones.
BUG Iterar o.aplicaciones en _buildWAMsg

🟡 Problemas de UX

06
Grid de productos fuerza scroll horizontal en mobile
La grilla de 7 columnas tiene min-width: 460px (linea 299). En un celular de 375px el usuario debe scrollear para ver Vol, $/ha y Monto. Estos datos son criticos para el agronomo.
MOBILE Diseño card-stack o colapsar columnas
07
No hay boton "Eliminar" visible en ordenes
deleteOrden() existe (linea 1844) pero ningun boton del UI lo invoca. El usuario no puede eliminar ordenes erroneas.
UX Agregar boton eliminar en card de orden
08
PDF de orden guardada: hack de swap de form values
exportarOrdenPDF() (linea 1651) hace un swap temporal de los valores del formulario para reusar exportarPDF(). Fragil y propenso a bugs si el form esta en estado de edicion.
TECH DEBT exportarPDF deberia recibir un objeto orden
09
Campaña se edita con prompt()
Linea 596. Mismo problema que campos. Deberia ser un input inline o un select con opciones predefinidas (2024/25, 2025/26, 2026/27).
MENOR Select inline
10
No hay confirmacion al salir con OT sin guardar
Si el agronomo esta armando una OT con 5 productos y toca otro tab o cierra el browser, pierde todo sin aviso.
UX beforeunload + confirm en cambio de tab
11
No hay totales visibles en Nueva OT
El "Resumen retiro" solo muestra volumenes, no costos. El usuario no ve cuanto va a costar la aplicacion hasta que guarda y va a Analisis.
UX Agregar footer de totales (como en cotizador agro)

🟢 Lo que funciona bien

  • Multi-aplicacion: El sistema de multiples lotes por OT esta bien implementado con la estructura aplicaciones[]
  • Dropdown de productos: Busqueda con keyboard nav (flechas + Enter), categorias, agregar manual. Bien resuelto.
  • Status buttons: Cambiar estado (Pend/Curso/Comp) directo desde la card sin entrar a editar. Excelente para uso rapido en campo.
  • Catalogo desde Sheets: Los precios y productos se actualizan sin redesplegar.
  • Aplicadores memorizados: Se guardan y sugieren en futuras OTs.
  • Tipos de labor personalizables: Con defaults sensatos + opcion de agregar.
  • Design system coherente: Mismas variables CSS, tipografias y componentes que los cotizadores.
  • PDF profesional: Buena estructura con header navy, tabla por lote, resumen retiro.

🔧 Plan de optimizacion propuesto

Fase 1 — Fixes criticos (impacto inmediato)

1
Modal inline para Campos/Lotes — Reemplazar los 6-7 prompt() por un modal con form completo (nombre, cliente, localidad, lotes con has y cultivo). El usuario ve todo junto, puede corregir antes de guardar. Mobile-friendly.
2
Sanitizar HTML — Crear funcion esc(str) y usarla en todo innerHTML. Previene XSS con datos de usuario.
3
Export/Import JSON — Boton "Descargar backup" (JSON) y "Restaurar" (file input). Cero dependencia de servidor, el usuario tiene su archivo.
4
Fix Analisis multi-app — Iterar o.aplicaciones en vez de usar campos legacy. Corrige conteo de hectareas y costos.

Fase 2 — UX mobile

5
Grilla responsive de productos — En mobile: card-stack (producto + dosis + costo apilados) en vez de tabla con scroll horizontal. Mantener tabla en desktop.
6
Footer de totales en Nueva OT — Barra sticky con: insumos USD/ha, labor USD/ha, total USD/ha, total USD. Mismo estilo navy gradient que los cotizadores.
7
Boton eliminar en ordenes + confirmacion.

Fase 3 — Robustez

8
Fix WA multi-app — Mensaje que liste cada lote con sus productos y volumenes.
9
Refactor exportarPDF — Que reciba un objeto orden en vez de leer del form.

Proteccion de datos — beforeunload si hay OT en progreso. Selector de campana inline.

📋 Resumen ejecutivo

La app tiene una base solida (multi-aplicacion, catalogo Sheets, PDF, WA) pero la UX esta lastrada por el uso de prompt() para CRUD, la falta de responsividad en mobile, y bugs en Analisis/WA que no contemplan ordenes multi-lote.

Las 3 mejoras de mayor impacto son:

  1. Modal inline para campos/lotes — cambia drasticamente la experiencia en celular
  2. Footer de totales + grilla responsive — el agronomo necesita ver costos al armar la OT
  3. Export/Import JSON — protege los datos del usuario sin necesidad de backend
